跨域总结整理

发表于 2017-10-12 | 分类于 javascript |

前端的日常工作中经常会遇到跨域的问题，跨域到底是啥，又该怎么解决呢？

什么是跨域？

根据字面意思去理解，跨域就是跨域名，具体点说，只要协议、域名、端口有任何一个不同，都会被视作是不同的域。

广义的跨域：

1. 资源跳转： A链接、重定向、表单提交
2. 资源嵌入： <link>、<script>、<img>、<frame>等dom标签，还有样式中background:url()、@font-face()等文件外链
3. 脚本请求： js发起的ajax请求、dom和js对象的跨域操作等

在浏览器中，有个叫做同源策略的安全概念，同源策略/SOP（Same origin policy）是一种约定，是浏览器最核心也最基本的安全功能，如果缺少了同源策略，浏览器很容易受到XSS、CSFR等攻击。所谓同源是指”协议+域名+端口”三者相同，即便两个不同的域名指向同一个ip地址，也非同源。

从出于安全性考虑的角度来看，跨域的限制存在是挺合理的，但是这种安全限制另一方面却给注入iframe 或是ajax 应用上带来了不少麻烦。所以要通过一些方法使本域的js能够操作其他域的页面对象或者使其他域的js能操作本域的页面对象（iframe之间）。下面是具体的跨域情况详解：

跨域解决方法

  1. 通过jsonp跨域
  2. document.domain + iframe跨域
  3. location.hash + iframe跨域
  4. window.name + iframe跨域
  5. postMessage跨域
  6. 跨域资源共享（CORS）

jsonp 跨域

通过为减轻web 服务器的负载，把js、css、img 等静态资源分离到另一台独立域名的服务器上，在html 页面中再通过相应的标签从不同域名下加载静态资源，而被浏览器允许，基于这个原理，我们可以通过动态创建script，再请求一个带参数网址实现跨域通信。

document.domain + iframe 跨域

当主域相同，子域不同的时候，不同域的框架之间是不能进行js 的交互操作。不同框架之间是可以获取window 对象，但无法获取相应的属性和方法。

比如，有一个页面：www.somedomainyup.com/a.html，这个页面中有一个地址是somedomainyup.com/b.html 的iframe，因为两者是不同域，所以无法在页面中获取iframe 中的东西。

我们可以用document.domain 去解决这种跨域问题，实现原理是两个页面都通过js强制设置document.domain为基础主域，就实现了同域。

location.hash + iframe 跨域

因为父窗口可以对iframe 进行URL 读写，iframe 也可以读写父窗口的URL，URL 有一部分被称为hash，就是#号及其后面的字符，hash 一般用于浏览器锚点定位，而HTTP 的请求过程中不会携带hash，所以这部分的修改不会产生HTTP 请求，但是会产生浏览器历史记录。

此方法的原理就是改变URL 的hash 部分来进行双向通信。每个window 通过改变其他window 的location 来发送信息（由于两个页面不在同一个域下，IE、Chrome 不允许修改parent.location.hash 的值，所以要借助父窗口域名下的一个代理iframe），通过监听自己的URL 变化来接受消息。这这个方式的跨域通信有比较明显的缺点，比如会造成一些不必要的浏览器历史记录，有些浏览器不支持onhashchange 事件，需要轮询来获知URL 的改变。另外还有其他缺点就是数据会被直接暴露在URL 中，数据容量和类型都有限制等。

假如父页面是baidu.com/a.html，iframe 嵌入的页面为google.com/b.html，要实现此两个页面间的通信可以通过以下方法。

  • a.html 传送数据到b.html
    • a.html 下修改iframe 的src 为google.com/b.html#user=admin
    • b.html 监听到url 发生变化，触发相应操作
  • b.html 传送数据到a.html，由于两个页面不在同一个域下IE、Chrome 不允许修改parent.location.hash 的值，所以要借助于父窗口域名下的一个代理iframe
    • b.html 下创建一个隐藏的iframe ，此iframe 的src 是baidu.com 域下的，并挂上要传送的hash 数据，如src=”www.baidu.com/c.html#…”
    • c.html 监听到url 发生变化，传数据给a.html（因为a.html 和c.html 同域）
    • a.html 触发相应操作

window.name + iframe跨域

window 对象有个name 属性，该属性有个特征：即在一个窗口（window）的生命周期内，窗口载入的所有页面都是共享一个window.name 的值，每个页面对window.name 都有读写的权限，window.name 是持久存在一个窗口载入过的所有页面中的，并不会因为新页面的载入而进行重置。

postMessage跨域

前面提到的方法或多或少看起来都像是些hack 的手段，可以看出来以前大家为了跨域琢磨出了多少奇技淫巧，难道跨域就没有些“规范”点的办法吗？有的，postMessage是HTML5 XMLHttpRequest Level 2中的API，且是为数不多可以跨域操作的window属性之一，它可用于解决以下方面的问题：

  • 页面和其打开的新窗口的数据传递
  • 多窗口之间消息传递
  • 页面与嵌套的iframe 消息传递
  • 上面三个场景的跨域数据传递

高级浏览器Internet Explorer 8+, chrome，Firefox , Opera 和 Safari 都将支持这个API。发送消息的为’postMessage’方法，接受信息的为’message’事件。比如somedomain.com/a.html 通过iframe 嵌入了一个anotherdomain.com/b.html页面，可以这样实现互相通信

postMessage 的使用方法：

  • otherWindow.postMessage(message, targetOrigin)
    • otherWindow：目标窗口
    • message：要发送的消息，类型为String、Object（IE8、9不支持），部分浏览器只支持字符串，所以传参时最好用JSON.stringify()序列化。
    • targetOrigin：限定消息接受范围，协议+主机+端口号，设置为”*”，表示可以传递给任意窗口，如果要指定和当前窗口同源的话设置为”/“。

跨域资源共享（CORS）

CORS也是一种较为普遍的规范的跨域方法，CORS是一个W3C标准，全称是”跨域资源共享”（Cross-origin resource sharing）。

它允许浏览器向跨源服务器，发出XMLHttpRequest请求，从而克服了AJAX只能同源使用的限制。CORS背后的基本思想就是使用自定义的HTTP头部让浏览器与服务器进行沟通，从而决定请求或响应是应该成功还是失败。目前，所有浏览器都支持该功能，IE浏览器不能低于IE10(IE8+：IE8/9需要使用XDomainRequest对象来支持CORS)。整个CORS通信过程，都是浏览器自动完成，不需要用户参与。对于开发者来说，CORS通信与同源的AJAX通信没有差别，代码完全一样。浏览器一旦发现AJAX请求跨源，就会自动添加一些附加的头信息，有时还会多出一次附加的请求，但用户不会有感觉。

实现CORS通信的关键是服务器。只要服务器实现了CORS接口，就可以跨源通信。
